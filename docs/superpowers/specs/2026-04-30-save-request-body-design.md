# 统计功能保存原始请求体 — 设计方案

## 背景

`llm-plug` 的统计功能目前将请求元数据（模型、通道、延迟、token 数、headers 等）持久化到 PostgreSQL 的 `requests` 表中，但**不保存原始请求体**（即用户实际发送的 messages/prompt 内容）。

原始请求体仅在 `DEBUG=true` 时写入本地磁盘的 JSONL 日志文件，这无法满足审计和长期分析的需求。

## 需求

| 项目 | 说明 |
|------|------|
| **目的** | 审计 + 分析 |
| **保存范围** | 全部请求（成功与失败） |
| **保留策略** | 永久保存 |
| **请求量级** | 约 10,000 次/天 |
| **请求体大小** | 通常几 KB ~ 几百 KB（取决于上下文长度，多模态含 base64 图片时可能更大） |

## 方案对比

### 方案 A：直接存入 PostgreSQL

在现有 `requests` 表中增加 `request_body` 列（类型 `TEXT` 或 `JSONB`），`stats.record_request()` 写入请求时一并存入原始请求体字符串。

#### 实现逻辑
1. **表结构变更**：`ALTER TABLE requests ADD COLUMN request_body JSONB`
2. **写入**：`proxy_core.py` 中调用 `record_request()` 时传入 `request_body=原始请求体字符串`
3. **查询**：`GET /admin/requests` 及其详情接口直接 `SELECT request_body FROM requests WHERE id = ?`
4. **前端**：请求详情弹窗中新增 "请求体" 标签页或折叠面板展示原始 JSON

#### 优势
- 实现最简单，只需改表结构 + 一两个函数签名
- 查询极方便，一条 SQL 即可关联元数据和请求体
- 事务一致性天然保证（元数据和请求体同成功/同失败）

#### 劣势
- 表体积随时间线性膨胀。按平均 20KB/请求估算：
  - 200 MB/天，6 GB/月，72 GB/年
  - 若含多模态 base64 图片，可能数倍于此
- `requests` 表变宽后，前端列表页查询（不需要请求体时）也会扫描更宽的行，拖慢性能
- PostgreSQL 备份、迁移、索引维护成本上升
- 大字段（TOAST）在频繁更新/查询时性能下降

---

### 方案 B：PostgreSQL + 本地文件系统

`requests` 表仅增加一个 `body_path` 字符串列，原始请求体以独立文件形式存储到本地磁盘，PG 只存文件路径。

#### 实现逻辑
1. **表结构变更**：`ALTER TABLE requests ADD COLUMN body_path VARCHAR(512)`
2. **存储目录结构**：
   ```
   data/request_bodies/
   └── 2026/
       └── 04/
           └── 30/
               ├── <uuid1>.json
               ├── <uuid2>.json
               └── ...
   ```
3. **写入流程**：
   - `proxy_core.py` 在调用 `record_request()` 之前（或同时）
   - 将原始请求体序列化为 JSON 字符串
   - 写入文件 `data/request_bodies/YYYY/MM/DD/<request_id>.json`
   - 将文件路径传给 `record_request()` 存入 `body_path`
4. **查询流程**：
   - `GET /admin/requests` 列表接口：不读取文件，仅返回 `body_path`（或干脆不返回，只在前端详情页按需加载）
   - 详情接口：根据 `body_path` 读取文件内容返回
5. **清理/归档**：文件系统层面可独立压缩旧目录（如 `.tar.gz`），不影响 PG 表性能

#### 优势
- `requests` 表保持轻量，元数据查询性能不受影响
- 文件系统天然适合存储大对象（base64 图片等不会拖垮 PG 行）
- 可单独对旧文件进行压缩、归档、迁移，不影响在线查询
- 备份可分两部分：PG 备份（小）+ 文件目录备份（可增量）
- 存储成本低于同等容量的数据库存储

#### 劣势
- 需要管理文件生命周期（磁盘满监控、权限、清理）
- 读取请求体需要一次文件 IO，比 PG 内查询稍慢（但对本地 SSD 来说可忽略）
- 分布式/容器化部署时，多实例需要共享同一个文件目录（通过 NAS/共享卷），否则部分实例读不到文件
- 文件与数据库记录之间的一致性需要应用层维护（文件写成功但 DB 事务回滚时，会留下孤儿文件）

---

### 方案 C：PostgreSQL + 对象存储（S3/MinIO）

与方案 B 类似，但将请求体存储到对象存储服务（如 AWS S3、阿里云 OSS、或自建的 MinIO），PG 中保存对象的 `bucket/key`。

#### 实现逻辑
1. **表结构变更**：`ALTER TABLE requests ADD COLUMN body_s3_key VARCHAR(512)`
2. **写入流程**：
   - 请求到达后，生成唯一 key：`request-bodies/YYYY/MM/DD/<request_id>.json`
   - 异步或同步将请求体上传至对象存储
   - 将 key 存入 `requests.body_s3_key`
3. **查询流程**：
   - 详情接口根据 key 向对象存储发起 `GetObject` 请求，返回内容
   - 或生成预签名 URL 让前端直接访问对象存储（减少服务端带宽）
4. **生命周期**：利用对象存储的生命周期策略自动转冷存、压缩或归档

#### 优势
- 扩展性最强，几乎无限容量，无需关心磁盘满问题
- 可利用对象存储的压缩、生命周期管理、跨区域复制等高级特性
- 与 PG 完全解耦，数据库压力最小
- 预签名 URL 可直接让浏览器从对象存储下载，不占用服务端带宽
- 天然适合多机/容器化/K8s 环境，无需共享卷

#### 劣势
- 引入额外基础设施依赖（哪怕是 MinIO 也需要部署和维护）
- 每次读取请求体都是网络请求（比本地文件慢，但 10K/天的访问量下完全可接受）
- 本地开发环境需要配置对象存储或模拟服务
- 实现复杂度最高（需要引入 S3 SDK、处理上传失败/重试、权限配置等）
- 离线审计时（如直接连 PG 查数据），需要额外网络才能获取请求体

## 方案评估矩阵

| 维度 | 方案 A (纯 PG) | 方案 B (PG + 文件) | 方案 C (PG + S3) |
|------|---------------|-------------------|-----------------|
| **实现复杂度** | 低 | 中 | 高 |
| **查询便捷性** | 极高 | 高 | 中 |
| **元数据查询性能** | 中（表膨胀后下降） | 高（表始终轻量） | 高 |
| **存储成本** | 高（数据库存储贵） | 低 | 中（对象存储便宜但可能有流量费） |
| **扩展性** | 低 | 中 | 高 |
| **容器化/分布式友好** | 高 | 低（需共享卷） | 高 |
| **备份复杂度** | 低（单份备份） | 中（DB + 文件） | 中（DB + 对象存储） |
| **离线可用性** | 高 | 高 | 中（需网络访问 S3） |
| **长期维护成本** | 高（表越来越大） | 低 | 低 |

## 推荐

- **当前阶段（单机/小集群）**：**方案 B（PG + 文件系统）** 是最佳平衡点。实现成本可控，PG 表不膨胀，文件系统天然适合永久归档，且 1万/天的写入量对任何现代磁盘都是九牛一毛。
- **未来演进**：如果项目走向大规模分布式部署，或团队已有对象存储基础设施，可平滑迁移到方案 C——将文件同步到 S3 后把 `body_path` 改为 `body_s3_key` 即可。
- **不推荐方案 A**：除非请求体极小（<1KB）且总量可控，否则长期来看表膨胀会带来难以预料的性能和维护问题。

## 待决策事项

在正式开始实现前，还需明确以下几点：

1. **前端展示方式**：请求体在列表页是否展示？还是仅在详情页按需加载？（强烈建议仅详情页展示，避免列表页加载大量数据）
2. **是否压缩**：文件/对象存储层面是否启用 gzip 压缩？（可节省 50%~80% 空间，对 JSON 文本效果显著）
3. **权限控制**：原始请求体可能包含敏感信息（API key、个人隐私数据），是否需要额外的访问控制或脱敏策略？
4. **清理策略补充**：虽然需求是永久保存，但是否需要“热/冷分离”策略？例如最近 30 天存本地/对象存储标准层，更早的转冷存/归档？
